# SLURM Orchestration (Two-Stage Experiments 1–3)

This directory contains **cluster execution scripts** for the Input Representation Benchmark.

## Design principles

- **Determinism separation**: tokenization and MEDS-derived outcome extraction are **deterministic** given inputs (MEDS parquet + tokenizer config), so they should be executed **once per configuration** (not once per seed).
- **Avoid write races**: tokenization writes large shared artifacts; parallel SLURM execution must avoid multiple tasks writing the same output directories.
- **Reproducibility**: each stage is a single-purpose script with explicit inputs/outputs and stable filenames.

## Experiments use a two-stage architecture

Across Exp1–Exp3, **Stage 0** is the deterministic “data materialization” step (tokenize + outcomes),
and **Stage 1** is the stochastic “model training” step (train + classify).

- **Stage 0** depends on: data + tokenizer config + quantizer/anchoring/fused/temporal.
- **Stage 1** depends on: the above **plus** the training seed and training hyperparameters.

This separation avoids redundant work and prevents SLURM array tasks from racing to write the same
`<data_version>-tokenized/` directories.

### Experiment 1 (Exp1): Granularity

### Stage 0 (CPU-heavy): tokenize + outcomes (one job per config)

- **Script**: `03_stage0_tokenize_and_outcomes_meds.sh`
- **Jobfile (one-line commands)**: `04_exp1_tokenize_jobs.sh` (generated by `run_experiments.py`)
- **Runner**: `02_run_from_jobfile_cpu.sh` (SLURM array runner; **no GPU request**)

Submit (recommended conservative concurrency due to RAM/IO):

```bash
sbatch --array=0-11%2 slurm/02_run_from_jobfile_cpu.sh slurm/04_exp1_tokenize_jobs.sh
```

Outputs:

- Tokenized timelines:
  - `${DATA_DIR}/data/<data_version>-tokenized/{train,val,test}/tokens_timelines.parquet`
  - `${DATA_DIR}/data/<data_version>_first_24h-tokenized/{train,val,test}/tokens_timelines.parquet`
- Outcomes joined onto tokenized timelines (written into the `_first_24h-tokenized` dirs):
  - `${DATA_DIR}/data/<data_version>_first_24h-tokenized/{train,val,test}/tokens_timelines_outcomes.parquet`

Implementation detail: Stage 0 can optionally write tokenized artifacts into a shared cache root
(`IRB_TOKEN_CACHE_ROOT`) and symlink them back into `${DATA_DIR}/data/…` to avoid duplication.

### Stage 1 (GPU): pretrain + classify (one job per config×seed)

- **Script**: `06_exp1_stage1_train_and_classify.sh`
- **Jobfile (one-line commands)**:
  - demo: `07_exp1_train_jobs_demo.sh` (generated)
  - full: `07_exp1_train_jobs_full.sh` (generated)
- **Runner**: `run_from_jobfile.sh` (GPU array runner)

Submit (max 8 concurrent GPUs):

```bash
sbatch --array=0-11%8 slurm/run_from_jobfile.sh slurm/07_exp1_train_jobs_demo.sh
```

Stage 1 **assumes Stage 0 has completed** and therefore never re-runs tokenization.

## Common infrastructure

- **Environment**: `preamble.sh` activates the conda env and exports:
  - `IRB_HOME`, `FMS_EHRS_HOME`, `DATA_DIR`, `MODEL_DIR`, `OUTPUT_DIR`
- **Array runners**:
  - `02_run_from_jobfile_cpu.sh`: executes the Nth non-empty, non-comment line from a jobfile
  - `run_from_jobfile.sh`: same, but requests `--gres=gpu:1`

## Partitions

This repo defaults to `gpuq` in SBATCH headers because our primary target is the randi GPU cluster.
If your site uses different CPU/GPU partitions, override at submit time, e.g.:

```bash
sbatch --partition=<your_cpu_partition> --array=0-11%2 slurm/02_run_from_jobfile_cpu.sh slurm/04_exp1_tokenize_jobs.sh
```
